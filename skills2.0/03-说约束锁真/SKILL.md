# 03 - 【说】约束锁真 Skill

> **墨学经说双链推理架构** 说层模块，架构护栏核心。说者，锁定词义、边界、规则。

## 核心定位

《墨经》之"说"：对"经"的极简命题进行定向锁义，消除歧义、锁定边界。本模块将其标准化为**固定三锁结构 + 智能松紧适配**的约束生成流程。

**降温 ≠ 死板**：降温是锁定语义边界，但在发散模式下应保留合理创意空间。

## 一、输入

**双源对照**（杜绝失真）：

```json
{
  "source_1": "用户原始清洗文本",
  "source_2": "自检通过的经文（或降级标记）",
  "preserved_conditions": ["条件1", "条件2"],
  "preserved_prohibitions": ["禁止项1"],
  "conflicts_carried": [],
  "degradation_triggered": false
}
```

> 双源对照原则：以用户原文和经文双重对照生成约束，任一源的信息都不可忽略。

## 二、固定三锁结构（强制必填）

### 2.1 词义锁

对高频歧义字**强制逐条唯一释义**，排除所有引申义。

**内置高频歧义词黑名单**（命中必锁）：

```
歧义词表：
道、势、权、治、损益、自治、机、变、可爱、杀伐、
自然、根本、适当、合理、相关、基本、主要、一般、
正常、特殊、简单、复杂、重要、紧急、优先、高级、
智能、安全、高效、稳定、灵活、兼容、统一、标准
```

**词义锁格式**：

```
【说-词义锁】
- [歧义词]：[本条唯一释义]，排除[引申义1]、[引申义2]
- [歧义词]：[本条唯一释义]，排除[引申义1]
```

**示例**：

```
【说-词义锁】
- 可爱：柔和配色、圆润组件、轻量卡通元素，非酷炫、商务、暗黑风格
- 青少年：适配低龄审美，简洁清爽、无复杂视觉压迫，非成人化、非低幼化
```

### 2.2 边界锁

明确**允许维度**和**严格禁止维度**。

**边界锁格式**：

```
【说-边界锁】
- 允许推演维度：[维度1]、[维度2]、[维度3]
- 禁止推演维度：[维度1]、[维度2]
```

**示例**：

```
【说-边界锁】
- 允许推演维度：网页UI、布局、配色、组件风格
- 禁止推演维度：后端架构、运维部署、备案流程、营销内容
```

### 2.3 规则锁

设定**推理底线、防编造、输出规范、推演规模上限**。

**规则锁格式**：

```
【说-规则锁】
- 推理底线：[底线规则]
- 防编造：[防编造规则]
- 输出规范：[输出格式要求]
- 推演规模上限：[最大方案数/最大维度数]
```

**示例**：

```
【说-规则锁】
- 推理底线：所有样式需可前端落地，不虚构不存在组件
- 防编造：引用技术框架须为真实存在的框架
- 输出规范：每方案含配色方案、组件列表、布局描述
- 推演规模上限：最多3套方案，每方案不超过5个核心组件
```

## 三、智能松紧适配

根据前置意图预判（由路由模块提供，或本模块预判），自动调整约束强度：

| 预判模式 | 词义锁 | 边界锁 | 规则锁 |
|----------|--------|--------|--------|
| 发散优先 | 严锁歧义 | 适度宽松，保留创意空间 | 仅底线护栏，不卡死可能性 |
| 收敛优先 | 全维度严锁 | 全维度严锁 | 全维度严锁，零脑补 |
| 混合审慎 | 严锁歧义 | 中等收紧 | 底线 + 关键约束 |

**发散模式适配示例**：

```
词义锁：严锁（可爱≠商务，必须明确）
边界锁：适度宽松（允许探索网页风格的多种可爱变体：萌系、治愈系、活力系）
规则锁：仅底线（所有方案需可落地，不虚构组件），不限制创意方向
```

**收敛模式适配示例**：

```
词义锁：全维度严锁（每个关键词唯一释义）
边界锁：全维度严锁（仅允许查询指定数据，禁止延伸分析）
规则锁：零脑补（数据不足如实留白，禁止任何推断填补）
```

## 四、约束自校验

生成约束后，自动检测约束内部冲突：

| 检查项 | 判定标准 |
|--------|----------|
| 边界自相矛盾 | 允许维度与禁止维度有重叠 → **冲突** |
| 词义与边界矛盾 | 词义锁限定的范围与边界锁矛盾 → **冲突** |
| 规则与边界矛盾 | 规则锁要求超出边界锁允许范围 → **冲突** |
| 松紧适配矛盾 | 松紧度与预判模式不一致 → **冲突** |

**冲突处理**：检测到冲突则**重构重生成**约束，最多重试 2 次。

## 五、冲突透传处理

当预处理模块标记了语义冲突时，在词义锁中专项处理：

```
冲突标记：[[CONFLICT:STYLE]] 可爱 vs 商务
词义锁处理：
- 可爱：[释义]，与商务严谨风格存在冲突，本条按用户优先顺序处理
- 商务严谨：[释义]，与可爱风格存在冲突，需在方案中分别呈现或融合处理
```

## 六、输出规范

```json
{
  "gloss_lock": "词义锁全文",
  "boundary_lock": "边界锁全文",
  "rule_lock": "规则锁全文",
  "tightness_mode": "loose | medium | strict",
  "self_check": {
    "passed": true,
    "conflicts_found": []
  },
  "full_constraint_text": "【说-词义锁】...\n【说-边界锁】...\n【说-规则锁】..."
}
```

## 七、执行流程

```
双源输入（用户原文 + 经文）+ 预判模式
    │
    ▼
[1] 歧义词扫描：匹配高频歧义词黑名单 + 识别上下文歧义词
    │
    ▼
[2] 生成词义锁：逐条唯一释义，排除引申义
    │── 有冲突标记 ──▶ 在词义锁中专项处理冲突
    │
    ▼
[3] 生成边界锁：基于双源提取允许/禁止维度
    │
    ▼
[4] 生成规则锁：设定底线、防编造、输出规范、规模上限
    │
    ▼
[5] 松紧适配：根据预判模式调整约束强度
    │
    ▼
[6] 约束自校验：检测内部冲突
    │── 有冲突 ──▶ 重构重生成（最多2次）
    │
    ▼
[7] 输出三锁全文 + 松紧模式 + 自检结果
    │
    ▼
  交付【动态意图路由】模块（如未预判）或直接进入推理执行
```
